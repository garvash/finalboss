## Overview
本エピックでは、ゲームの物理的な基盤（勇者とモンスターの生成、移動、衝突）を実装する。左端（町）から湧き出る勇者と、右端（魔王城）からプレイヤーが召喚するモンスターが、一本道の通路で衝突し、互いに足止め（摩擦）を行う。Godot 4の `Area2D` と `Signals` を活用し、物理演算に頼らない確実な挙動を目指す。

## Objectives and Scope
- **In-Scope:** 勇者とモンスターのベースクラス作成、自動生成（スパウナー）、移動ロジック、衝突検知、玉座への到達判定。
- **Out-Of-Scope:** イカサマUI（Epic 2）、ゴールド消費（Epic 3）、アニメーションの詳細（Epic 3）。

## System Architecture Alignment
- `units/BaseUnit.gd` を共通親クラスとする `Area2D` ベースの構成。
- `area_entered` シグナルで足止めを制御し、`GlobalSignals` を介して玉座への到達（敗北）を通知。
- `GameManager (Autoload)` による集中状態管理。

## Detailed Design

### Services and Modules
- **`UnitSpawner`**: 勇者（左端）とモンスター（右端）の自動・手動生成を担当。
- **`BaseUnit` (Area2D)**: 全ての移動エンティティの基底クラス。移動速度、現在の状態（移動中/停止中）を管理。
- **`Hero` / `Monster`**: `BaseUnit` を継承。移動方向（右/左）と固有の衝突フィルタリングを持つ。
- **`CollisionController`**: `area_entered` シグナルをフックし、`Timer` を用いて「足止め時間」を制御。

### Data Models and Contracts
- **`UnitStats` (Resource)**: `speed: float`, `health: int`, `stop_time: float`。
- **`GameState` (Autoload)**: 現在の生存時間と玉座のHP（または敗北フラグ）を管理。

### APIs and Interfaces
- `GlobalSignals.hero_spawned(unit_id)`
- `GlobalSignals.monster_spawned(unit_id)`
- `GlobalSignals.hero_reached_throne()`

### Workflows and Sequencing
1. **生成**: `Spawner` が `Hero.tscn` をインスタンス化。
2. **行進**: `Hero` が `_process` 内で `x` 座標を移動。
3. **衝突**: `Monster` の `Area2D` と接触 → 双方の `speed` を 0 にし、`stop_timer` を開始。
4. **解決**: タイマー終了後、互いにダメージを与え、生存していれば `speed` を元に戻す。

## Non-Functional Requirements

### Performance
- 画面上に合計50体存在しても 60FPS を維持。

### Reliability/Availability
- 物理演算を使用せず、衝突時のオフセット調整でスタックを防止。

### Observability
- ユニットの生成、衝突、消滅の各イベントを `GlobalSignals` でログ出力可能にする。

## Dependencies and Integrations
- **Godot 4.6.1 Stable Engine**
- **Kenney UI/Pixel Art Assets**

## Acceptance Criteria (Authoritative)
1. 勇者が左から右へ、モンスターが右から左へ移動すること。
2. 両者が接触した際、即座に停止し、一定時間の「足止め」が発生すること。
3. 足止め時間終了後、一方が消滅（HP 0）するか、双方が移動を再開すること。
4. 勇者が右端の玉座に到達した際、ゲームオーバーのシグナルが発火し、画面に通知が表示されること。

## Risks, Assumptions, Open Questions
- **Risk:** ユニットの重なりが激しく、画面が混雑して視認性が落ちる可能性。
- **Assumption:** Kenneyアセットが即座に利用可能であること。
- **Question:** 勇者が一度に何体まで出現可能にするか（バランス調整）。

## Test Strategy Summary
- **Unit Test:** 各ユニットの `speed` リセット機能。
- **Integration Test:** `Spawner` による連続生成と、複数の衝突イベントの同時発生。
- **Edge Case:** 勇者がモンスターを追い越す、または玉座に同時に複数到達するケース。
